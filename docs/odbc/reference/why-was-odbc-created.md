# <a name="why-was-odbc-created"></a>ODBC가 만들어진 이유
지금까지 회사는 단일 DBMS를 사용 합니다. 데이터베이스에 대 한 모든 액세스는 해당 시스템에만 작동 하도록 작성 된 응용 프로그램 또는 해당 시스템의 프런트 엔드를 통해 수행 되었습니다. 그러나 컴퓨터의 사용 증가 자세한 컴퓨터 하드웨어 및 소프트웨어를 사용할 수 있게 되었습니다 회사 다른 Dbms를 획득 하기 시작 했습니다. 이유는 여러 다음과 같습니다. 있도록 이미 던 저렴, fastest, 던 구입 하는 사용자가 알고 된 단일 응용 프로그램에 대 한 가장 잘 작동 새로운 시장의 최신 내용입니다. 다른 이유는 단일 DBMS를 이전에 있던 부서 했 고 여러 재구성 및 합병 있었습니다.  
  
 문제는 개인용 컴퓨터 도입 되면서 더욱 복잡 하 게 증가 합니다. 다양 한 쿼리, 분석 및 데이터를 저렴 하 고 사용 하기 쉬운 데이터베이스의 수와 함께 표시에 대 한 도구에 이러한 컴퓨터입니다. 에 단일 기업 데스크톱, 서버 및 미니 수많은 분산, 다양 한 호환 되지 않는 데이터베이스에서에서 저장 및 모든 데이터를 가져올 수 있는 몇 가지 다양 한 도구를 다양 한 액세스를 경우가 많았습니다.  
  
 마지막 문제 컴퓨터 리소스를 가장 효율적으로 사용 하는 클라이언트/서버 컴퓨팅의 출현와 함께 제공 합니다. 저렴 한 개인용 컴퓨터 (클라이언트) 데스크톱에 앉아서 모두 그래픽 프런트 엔드를 데이터와 다양 한 스프레드시트, 프로그램 및 보고서 작성기 차트와 같은 저렴 한 도구를 제공 합니다. 미니 및 메인프레임 컴퓨터 (서버)는 사용할 수 있는 중앙 위치와 컴퓨팅 성능 신속 하 고 조정 된 데이터 액세스 권한을 제공 하는 Dbms를 호스트 합니다. 백 엔드 데이터베이스에 연결 해야 하는 프런트 엔드 소프트웨어어떻게?  
  
 유사한 문제가 독립 소프트웨어 공급 업체 (Isv)에 직면합니다. 미니 및 메인프레임 데이터베이스 소프트웨어를 작성 하는 공급 업체에 액세스 하고자 하는 각 DBMS에 대 한 DBMS 관련 코드를 작성 하거나 각 DBMS에 대 한 응용 프로그램의 버전을 작성 해야 일반적으로 했습니다. 개인용 컴퓨터에 대 한 소프트웨어를 작성 하는 공급 업체를 사용 하려면 각 DBMS에 대 한 데이터 액세스 루틴을 작성 해야 했습니다. 이 종종 막대 한 양의 리소스 작성 하 고 데이터 액세스 응용 프로그램 보다는 루틴 및 품질에 있지만 지정된 하는 DBMS의 데이터를 액세스할 수 있는지 여부에 응용 프로그램 판매 종종 유지 관리 하는 데 소요 된 의미 합니다.  
  
 개발자의 두 집합에 필요한 다양 한 Dbms의 데이터에 액세스 하는 방법을 했습니다. 메인프레임 및 미니 컴퓨터 그룹에는 개인용 컴퓨터 그룹에 필요한 모든 하나 DBMS의 독립 된 단일 응용 프로그램을 작성 하는 방법 뿐만 아니라이 기능 하는 동안에 단일 응용 프로그램에서 다양 한 Dbms의 데이터를 병합 하는 방법이 필요 합니다. 두 그룹의 데이터에 액세스 하는 상호 운용 가능한 방법은 필요한 즉, 데이터베이스 연결을 열 필요 합니다.
