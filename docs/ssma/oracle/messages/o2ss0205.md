# <a name="o2ss0205-unable-to-convert-pragma-autonomus_transaction-error"></a>O2SS0205: PRAGMA AUTONOMUS_TRANSACTION를 변환할 수 없습니다 (오류).

이 문서는 Oracle 용 SSMA (SQL Server Migration Assistant)가 블록 수준에서 문을 포함 하 `PRAGMA AUTONOMOUS_TRANSACTION` 는 PL/SQL 블록 (예: 독립 실행형 `BEGIN ... END`)을 변환 하지 않는 이유에 대해 설명 합니다.

## <a name="background"></a>배경

*자치 트랜잭션을* 사용 하면 부모 트랜잭션과 관계 없이 변경 내용을 커밋하거나 롤백할 수 있는 새 하위 트랜잭션을 만들 수 있습니다. `PRAGMA`는 컴파일러 지시문을 참조 하며 컴파일러에 명령을 제공 하는 데 사용 됩니다.

Oracle에서는 PL/SQL 블록을 *자치 트랜잭션으로*정의할 때 호출자의 트랜잭션 컨텍스트에서 해당 블록의 DML을 격리 합니다. 블록은 기본 트랜잭션 이라고 하는 다른 트랜잭션에 의해 시작 된 독립적인 트랜잭션이 됩니다. PL/SQL 블록을 자치 트랜잭션으로 표시 하려면 선언 섹션에 다음 문을 포함 하기만 하면 됩니다.

```sql
PRAGMA AUTONOMOUS_TRANSACTION;
```

위의 문에서는 `PRAGMA` `BEGIN` 전체 트랜잭션에 영향을 주지 않고 및 `END` 문 간에 PL/SQL 블록 내에서 자치 트랜잭션을 수행할 수 있습니다.

SSMA는 문을 사용 `PRAGMA AUTONOMUS_TRANSACTION` 하 여 (즉, 독립 실행형 `BEGIN ... END`) PL/SQL 블록을 변환 하려고 할 때마다 블록 `PRAGMA` 수준에서을 지원 하지 않습니다. 이는 SSMA가 SQL Server에서 해당 개체를 찾을 수 없기 때문입니다. 그러나 SSMA는 루틴 수준에서 자치 트랜잭션 (예: 프로시저, 함수 등)을 변환할 수 있습니다.

## <a name="example"></a>예제

다음과 같은 예제를 참조하세요.

```sql
DECLARE
    PRAGMA AUTONOMOUS_TRANSACTION;
BEGIN
    FOR i IN 3 .. 10 LOOP
        INSERT INTO at_test (id, description)
        VALUES (i, 'Description for ' || i);
    END LOOP;

    COMMIT;
END;
```

SSMA가 위의 PL/SQL 블록을 변환 하려고 시도 하면 다음 오류가 발생 합니다.

> O2SS0205: PRAGMA AUTONOMUS_TRANSACTION를 변환할 수 없습니다.

## <a name="possible-remedies"></a>가능한 해결 방법

`AUTONOMOUS_TRANSACTION` 문을 사용 하 여 PL/SQL 블록을 변환 하려면 코드를 프로시저, 함수 또는 트리거로 래핑하는 것이 좋습니다. 위의 예에서는 `at_test` 테이블에 데이터를 자치 트랜잭션으로 삽입 하는 프로시저를 정의할 수 있습니다. 위의 문제에 대 한 해결 방법은 다음과 같습니다.

```sql
CREATE OR REPLACE PROCEDURE Insert_Data
AS
    PRAGMA AUTONOMOUS_TRANSACTION;
BEGIN
    FOR i IN 3 .. 10 LOOP
        INSERT INTO at_test (id, description)
        VALUES (i, 'Description for ' || i);
    END LOOP;

    COMMIT;
END;
```

## <a name="how-ssma-converts-an-autonomous-transaction"></a>SSMA가 자치 트랜잭션을 변환 하는 방법

위의 *자치 트랜잭션* 문을 변환 하기 위해 ssma는 확장 저장 프로시저 에뮬레이터 `xp_ora2ms_exec2_ex`를 사용 합니다. 별도의 프로시저로 자치 블록을 만든 후 SSMA는 아래와 같이 에뮬레이터를 `xp_ora2ms_exec2_ex` 사용 하 여이 프로시저를 호출 합니다.

```sql
BEGIN
    DECLARE
        @active_spid INT,
        @login_time DATETIME

    SET @active_spid = ssma_oracle.get_active_spid()
    SET @login_time = ssma_oracle.get_active_login_time()

    EXECUTE master.dbo.xp_ora2ms_exec2_ex
        @active_spid,
        @login_time,
        DB_NAME(),
        N'DBO',
        N'INSERT_DATA$IMPL',
        N'false'
END
```

는 `xp_ora2ms_exec2_ex` 내부적으로 현재 자격 증명을 사용 하 SQL Server의 동일한 인스턴스에 다시 연결 하 고 위의 샘플`INSERT_DATA$IMPL` 에서 지정 된 저장 프로시저를 실행 합니다. 별도의 연결 이므로 실행 된 저장 프로시저에 의해 수행 된 수정은 호출 트랜잭션과 별도로 커밋되거나 롤백될 수 있습니다.

SQL Server 하기 위해 새 연결을 열어야 하므로 이러한 변환은 네이티브 Oracle 구현에 비해 상대적으로 느립니다. 가능 하면 항상 *자치 트랜잭션*없이 코드를 다시 작성 하는 것이 좋습니다.

## <a name="related-conversion-messages"></a>관련 변환 메시지

* O2SS0292: PRAGMA AUTONOMUS_TRANSACTION 변환 됨
* O2SS0308: PRAGMA AUTONOMOUS_TRANSACTION 무시 됩니다.
